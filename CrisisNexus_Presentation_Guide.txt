SECTION 1 — THE ELEVATOR PITCH (30 SECONDS)
--------------------------------------------------------------------------------
CrisisNexus is a live, browser-based emergency command center for Hyderabad.
A civilian who witnesses a crisis taps "Report" or hits "SOS" — and within a
second, every volunteer and city manager logged in (on phone, tablet, or PC)
sees the same incident pinned on a map of Hyderabad with the exact coordinates,
suggested nearest fire/police/ambulance unit, photo, and the reporter's voice
description (in any language). A volunteer dispatches the right number of
police cars, fire trucks, and ambulances with one click — the system
automatically routes them from the closest station of each type, calculates
real per-vehicle ETAs, and shows everyone the units en route in real time.

No fake data. Everything you see in the demo is what users actually generated.


SECTION 2 — WHY THIS MATTERS (THE PROBLEM)
--------------------------------------------------------------------------------
* Hyderabad has 10+ million people, 7 fire stations, dozens of hospitals,
  thousands of police officers — and no single screen that shows them what is
  happening RIGHT NOW.
* In a flood, fire, or building collapse, citizens call 100/101/108 separately,
  details get lost, dispatchers can't see the big picture, and response time
  suffers.
* Most existing systems are internal-only (operator screens) or citizen-only
  (panic-button apps). Nothing connects the two sides on a shared live map.
* Crises don't speak English. Most people in Hyderabad reach for Telugu, Hindi,
  or Urdu under stress — voice input must understand them.

CrisisNexus closes that loop.


SECTION 3 — THE 60-SECOND DEMO SCRIPT (CHRONOLOGY)
--------------------------------------------------------------------------------
Open the URL on TWO browsers side-by-side. One picks "Civilian", one picks
"Volunteer/Manager".

  1. CIVILIAN side: Click "Report Crisis" → pick "Fire" → tap the map at
     Banjara Hills → tap the mic, say in Hindi or Telugu "बहुत धुआं है, तीन
     लोग फँसे हैं" → optional photo → Submit.

  2. VOLUNTEER side: Within ~200ms, a red P1 marker pulses on the map. A toast
     pops up. The "Online" pill ticks. Audit log records "CREATED".

  3. Volunteer clicks the marker → detail panel opens with photo, exact GPS,
     reporter's name, transcribed voice, casualties, and the closest
     responders pre-computed (closest fire station, closest hospital, etc).

  4. Volunteer uses the dropdowns: 2× Police, 1× Fire Truck, 3× Ambulance →
     "Confirm Dispatch". The system reads the closest station OF EACH TYPE,
     computes real km + ETA per vehicle, and broadcasts.

  5. Both screens now show: "2× Police from Banjara Hills Police — 0.7 km —
     ETA 0.7 min, EN ROUTE | 1× Fire Truck — 0.0 km — ETA 0.0 min | 3×
     Ambulance from 108 Central — 6.5 km — ETA 7.8 min".

  6. Volunteer marks Resolved → both screens update → audit log records the
     full dispatch summary and resolver name.

That entire flow happens with no page refreshes, no databases involved, no
mocked data — pure live WebSocket sync.

SECTION 5 — TECHNICAL ARCHITECTURE (FOR JUDGES WHO ASK)
--------------------------------------------------------------------------------
LANGUAGES & LIBRARIES
  * Backend: Python 3.11 + FastAPI + Uvicorn + native WebSocket protocol.
  * Geospatial engine: Plain C (geo_engine.c), compiled to a shared library
    (.so) at server boot via gcc, loaded into Python using ctypes. Functions:
    haversine distance, find_nearest_unit (linear scan over emergency
    services), estimate_eta_minutes (distance / speed).
  * Frontend: Vanilla JavaScript (no React/Vue overhead), Leaflet 1.9.4 for
    maps, Chart.js 4.4 for analytics, CARTO Dark map tiles, OpenStreetMap
    Nominatim for free Hyderabad address search.
  * Voice: Browser-native Web Speech API (window.SpeechRecognition) — zero
    server cost, supports 20+ languages including all major Indian languages.
  * Storage: In-memory Python lists (incidents, audit_log) — chosen
    deliberately so the demo is self-contained and resets cleanly. Postgres
    is a one-line swap if production-grade persistence is wanted.

REAL-TIME LAYER
  * Single FastAPI WebSocket endpoint `/ws?name=X&role=Y`.
  * ConnectionManager class tracks every connection with role and name.
  * On every state change (new incident, dispatch, resolve, delete, SOS,
    presence change), the manager broadcasts a JSON event to ALL connected
    clients. Clients diff the event into local state and re-render.
  * Auto-reconnect on disconnect every 2 seconds — survives network blips.

ROLES
  * Civilian: Report + SOS + their own dashboard. Cannot see command center,
    cannot dispatch, cannot delete.
  * Volunteer/Manager: Full command center, dispatch, resolve, delete, audit
    log, analytics.
  * Switching roles is instant — click your name in the header.

EMERGENCY SERVICES PRE-MAPPED (real Hyderabad coordinates)
  * 8 hospitals (NIMS, Apollo Jubilee Hills, CARE Banjara Hills, Yashoda
    Secunderabad, KIMS Kondapur, Continental, Osmania, Gandhi).
  * 5 fire stations (Secunderabad, Banjara Hills, Gachibowli, Old City,
    Kukatpally).
  * 5 police stations (Cyberabad HQ, Hyderabad Commissionerate, Banjara
    Hills, Secunderabad, Madhapur).
  * 4 ambulance hubs (108 Central, West, East, North).

ETA MODEL
  * Police: 55 km/h average urban speed (sirens, light cars).
  * Ambulance: 50 km/h (108 with sirens).
  * Fire trucks: 40 km/h (heavier vehicles, slower).
  * Hospital teams: 45 km/h.
  * ETA = distance_km / speed_kmh × 60 — computed in C, returned in minutes.

SECURITY / PRIVACY NOTES (mention if asked)
  * No login or password — chosen for emergency speed-of-use.
  * Reporter's name + optional contact are visible only to volunteers.
  * Photos are session-only, never persisted to disk.
  * SOS only broadcasts after explicit GPS permission grant.

SECTION 9 — ANTICIPATED QUESTIONS & ANSWERS
--------------------------------------------------------------------------------
Q: "Is this connected to the actual Hyderabad emergency services?"
A: Not yet. The 22 stations are real coordinates, but dispatch is a
simulation showing what the OS layer would look like. Integration with
Telangana Police 100 / Fire 101 / 108 Ambulance dispatch APIs is a one-week
engineering effort once we have access.

Q: "What if internet goes down?"
A: We auto-reconnect every 2 seconds. The frontend keeps the last known state
visible. For true offline, we'd add Service Worker caching and IndexedDB
queueing — straightforward addition.

Q: "How do you prevent fake/malicious reports?"
A: Currently volunteers can mark any report as deleted from the Active
Reports tab, and every deletion is audit-logged with the actor's name. Next
step is OTP verification on phone numbers and trust scoring per reporter.

Q: "Why no login?"
A: Speed of use. In an emergency, asking a citizen to remember a password
adds critical seconds. We chose a name + role onboarding (stored locally) so
the FIRST tap of the app brings them to "Report" or "SOS" with zero friction.
Optional Aadhaar/phone verification can layer on top.

Q: "Why C for the geo engine?"
A: Two reasons: (1) it's measurably faster than NumPy for small arrays
because of zero call overhead, and (2) it shows the team can drop into
systems languages when needed — not just glue Python libraries together.

Q: "How does it scale to 1 million concurrent users?"
A: The current single-process WebSocket design handles ~10K connections.
Beyond that we'd shard by district, use Redis pub/sub for cross-shard
broadcast, and put nginx in front. Standard scaling pattern, no rewrite
needed.

Q: "Privacy?"
A: No data is persisted to disk. Photos live in browser memory only.
Reporter contact info is visible only to logged-in volunteers. SOS requires
explicit GPS permission. Audit log records every access.

Q: "What did YOU build vs use a library for?"
A: We wrote: the entire backend (FastAPI app, WebSocket presence manager,
dispatch logic, audit system), the entire frontend (every page, modal,
chart, voice handler), and the C geospatial engine. We use: Leaflet (map
rendering), Chart.js (analytics), browser Speech API (voice), and CARTO map
tiles — all standard battle-tested open-source.
